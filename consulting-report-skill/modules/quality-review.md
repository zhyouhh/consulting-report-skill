# 报告质量审查模块

## 一、质量审查的重要性

质量审查是确保咨询报告专业性和准确性的关键环节，特别是对于"形式交付"场景。

## 二、三级审查体系

### 2.1 一级审查：内容准确性
**审查重点**：
- 数据准确性
- 逻辑一致性
- 事实可追溯性

**检查清单**：
- [ ] 所有数据是否标注来源
- [ ] 数据是否最新（标注时间）
- [ ] 计算是否正确
- [ ] 引用是否准确
- [ ] 结论是否有数据支撑

### 2.2 二级审查：结构完整性
**审查重点**：
- 章节结构
- 逻辑流程
- 内容完整性

**检查清单**：
- [ ] 是否有执行摘要
- [ ] 章节是否完整
- [ ] 逻辑是否连贯
- [ ] 是否符合模板要求
- [ ] 附录是否齐全

### 2.3 三级审查：形式规范性
**审查重点**：
- 格式规范
- 语言质量
- 视觉呈现

**检查清单**：
- [ ] 标题层级是否正确
- [ ] 字体格式是否统一
- [ ] 图表编号是否连续
- [ ] 页码是否正确
- [ ] 目录是否更新

## 三、专项审查要点

### 3.1 数据质量审查
- 数据来源可靠性
- 数据时效性
- 数据完整性
- 数据一致性（不同章节数据是否一致）

### 3.2 语言质量审查
- 去AI化检查（机械过渡词、空洞强调句）
- 专业术语准确性
- 语句通顺性
- 标点符号规范

### 3.3 图表质量审查
- 图表类型是否合适
- 数据标注是否清晰
- 图例是否完整
- 配色是否专业
- 图注是否准确

## 四、常见问题清单

### 4.1 内容问题
- ❌ 数据无来源
- ❌ 结论无支撑
- ❌ 逻辑跳跃
- ❌ 前后矛盾
- ❌ 建议不可执行

### 4.2 格式问题
- ❌ 标题层级混乱
- ❌ 字体不统一
- ❌ 图表编号错误
- ❌ 页码缺失
- ❌ 目录未更新

### 4.3 语言问题
- ❌ 机械过渡词（首先、其次、最后）
- ❌ 空洞形容词（非常、极其）
- ❌ 术语不准确
- ❌ 语句不通顺

## 五、审查流程

### 5.1 自审阶段
**执行人**：报告撰写人
**时间**：初稿完成后
**工具**：quality_check.sh 脚本

### 5.2 互审阶段
**执行人**：团队成员
**时间**：自审完成后
**重点**：逻辑和内容

### 5.3 终审阶段
**执行人**：项目负责人
**时间**：交付前
**重点**：整体质量和合规性

## 六、质量评分标准

### 6.1 内容质量（40分）
- 数据准确性（10分）
- 分析深度（10分）
- 逻辑严密性（10分）
- 建议可行性（10分）

### 6.2 结构质量（30分）
- 章节完整性（10分）
- 逻辑连贯性（10分）
- 重点突出性（10分）

### 6.3 形式质量（30分）
- 格式规范性（10分）
- 语言专业性（10分）
- 视觉美观性（10分）

**评分标准**：
- 90分以上：优秀
- 80-89分：良好
- 70-79分：合格
- 70分以下：需要修改

## 七、快速审查工具

使用 quality_check 脚本进行自动化检查：

```bash
# Linux/macOS
bash consulting-report-skill/scripts/quality_check.sh report.md

# Windows
powershell -ExecutionPolicy Bypass -File consulting-report-skill/scripts/quality_check.ps1 -FilePath report.md
```
