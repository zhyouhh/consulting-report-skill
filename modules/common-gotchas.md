# 常见错误清单（Common Gotchas）

> 每次写作前先过一遍这个清单。它的作用不是增加仪式感，而是提前避开那些最容易把咨询报告写“虚”、写“散”、写成后台沟通稿的问题。

## 1. 语言层面的高频失误

### 1.1 机械过渡词

错误示例：

```text
首先，分析市场现状。其次，评估竞争格局。最后，提出战略建议。
```

更好的写法：

```text
市场规模已达 500 亿元，年增长率保持在 15% 左右。竞争格局呈现一超多强态势，头部企业合计占据 60% 份额。基于这一结构，新增进入者更适合从细分场景切入，而不是直接参与价格战。
```

问题本质：不是“词不好”，而是没有真正形成判断链。

### 1.2 空洞形容词

错误示例：

```text
该企业具有非常强大的竞争优势，市场表现极其优异。
```

更好的写法：

```text
该企业市占率 35%，毛利率 42%，近三年营收复合增长率 32%，均显著高于行业均值。
```

问题本质：判断没有证据，只能靠形容词硬撑。

### 1.3 元叙事表达

错误示例：

```text
本章将从三个方面展开。下文重点讨论机制设计。以上内容为后文分析奠定基础。
```

更好的写法：

```text
机制设计需要同时解决组织协同、标准统一和运营闭环三个问题，其中组织协同是首要前提。
```

问题本质：正文在解释“自己准备怎么写”，而不是直接写内容。

### 1.4 后台推进词泄漏

错误示例：

```text
根据技术规范书和内部材料，本文形成如下判断。
```

更好的写法：

```text
结合现有业务情况、行业实践和公开资料研判，当前最关键的问题集中在治理责任、数据运营和价值转化三个方面。
```

问题本质：把协作过程写进了正式交付。

### 1.5 缺少来源标注

错误示例：

```text
2025 年中国新能源汽车销量达 800 万辆。
```

更好的写法：

```text
2025 年中国新能源汽车销量达 800 万辆（数据来源：中汽协，2026 年 1 月）。
```

问题本质：事实不可追溯，可信度会直接下滑。

## 2. 结构层面的高频失误

### 2.1 把模板当成通用真理

错误示例：

```text
所有报告都必须有执行摘要、附录、术语表。
```

更好的写法：

```text
结构要以已确认的大纲和交付场景为准。执行摘要、术语与定义、附加说明等模块按项目需要配置，而不是默认强行套入。
```

问题本质：把某个项目的格式习惯误当成所有报告的统一标准。

### 2.2 章节拆得过细

错误示例：

```text
4.1
4.1.1
4.1.1.1
```

但每个标题下面只有一两句空话。

更好的写法：

把真正有独立信息量的层级保留下来，把薄内容并回上一级，先补厚内容再决定是否继续下探。

### 2.3 章节顺序不成链条

错误示例：

```text
第一章：组织架构
第二章：市场规模
第三章：实施计划
第四章：竞争格局
```

更好的写法：

```text
先交代背景和问题，再做分析与判断，最后提出机制和举措。
```

问题本质：读者读不到稳定的推导路径。

### 2.4 只报事实，不提炼意义

错误示例：

```text
2025 年市场规模 500 亿元，年增长率 15%。
```

更好的写法：

```text
2025 年市场规模达到 500 亿元，年增长率约 15%，说明行业仍处在扩张窗口期，进入者仍有机会通过差异化定位获取份额。
```

问题本质：缺少 `So What`，报告只剩素材，没有洞察。

## 3. 数据层面的高频失误

### 3.1 引用过时数据

不要用明显过期的数据硬推当前结论。时间敏感指标优先找最近一期，并明确时间边界。

### 3.2 结论与数据脱节

如果数据表现良好，却得出“应立即收缩”的结论，或者数据表现疲弱，却得出“基础非常扎实”的判断，就说明逻辑链没有打通。

### 3.3 缺少对比基准

单写“毛利率 42%”意义有限，最好补齐行业均值、历史变化或目标水平。

## 4. 呈现层面的高频失误

### 4.1 图表与正文重复

图表负责展示结构和趋势，正文负责提炼洞察和动作。两者如果只是在重复同一组数字，会浪费篇幅。

### 4.2 图表编号不连续

交付前要统一检查图表和表格编号，避免出现跳号、重号、正文引用与图题不一致。

### 4.3 列表太碎

一个标题下面连续挂 5 个只有一句话的小点，读起来会像机器拆句。能并成段落时就并成段落。

## 5. 流程层面的高频失误

### 5.1 跳过需求对齐

未确认篇幅、对象、截止时间就直接开始写，后续几乎一定返工。

### 5.2 不建 `plan/`

复杂项目如果没有 `plan/` 和 `progress.md`，第二轮开始就容易丢上下文、丢决策。

### 5.3 质量检查放到最后

正确做法不是“写完再统一看”，而是写完一章就先清掉高风险项，减少尾盘返工。

### 5.4 可审草稿边界说不清

如果导出了 `docx`，必须同步说明这是 `可审草稿`，不等于最终终排版，以免用户误以为已经完成模板级交付。

## 6. 开写前快速自检

- [ ] 是否已经确认目标、篇幅、时间和场景
- [ ] 是否已经读取 `modules/writing-core.md`
- [ ] 是否已经避开元叙事、自我解释和后台推进词
- [ ] 是否已经判断哪些章节是必需、哪些是可选
- [ ] 是否已经明确是否需要输出 `可审草稿`
