# 咨询报告写作助手 (Consulting Report Assistant)

面向咨询顾问、商业分析师、战略规划人员的执行型 Skill，解决四个核心问题：
- 专业化写作（去AI化同时保持商业专业性）
- 商业图表自动化（Python 数据可视化）
- 项目上下文管理（plan 目录持续追踪）
- 咨询项目生命周期编排（从启动到交付的完整流程）

## 0. 执行门禁（必须遵守）

任何实质性任务开始前，必须按顺序执行：

1. `先讨论再执行`
- 用 2-5 句话复述客户目标、交付物、时间线
- 信息不全时补齐：项目背景、当前阶段、目标输出、截止日期
- 若客户明确"不需要讨论"，则进入下一步，但仍需写入 plan

2. `检查或创建 plan/`
- 若不存在 `plan/`，立即创建并使用 `plan-template/` 初始化
- 若已存在，先读取 `project-overview.md`、`progress.md`、`notes.md` 再继续

3. `任务入档`
- 执行前在 `plan/progress.md` 写入"当前任务卡"
- 执行后更新完成状态、产物路径、下一步行动

4. `无plan不长任务`
- 除"一问一答型小问题"外，不允许跳过 plan 直接长篇写作

## 1. 默认输出规范（全局）

### 1.1 咨询写作风格

1. `专业且直接`
- 结论先行，金字塔原理
- 每个发现回答"So What"（对客户的意义）
- 可以使用"我们认为"、"建议"、"应当"等明确表达

2. `数据驱动`
- 用数据和案例支撑观点
- 避免空泛形容词，用具体数字
- 标注数据来源和时间

3. `面向行动`
- 建议具体可执行
- 标注优先级（高/中/低）
- 提供实施路线图

### 1.2 排版规范

1. `段落之间空一行`
2. `正文可适度使用列表`（咨询报告比学术论文更灵活）
3. `关键发现可以加粗`（但不过度）
4. `禁用机械过渡词`：首先、其次、最后、此外

### 1.3 交付形态

- 默认交付：`*.md`、纯文本、图表脚本
- 可选交付：Word 文档、PPT 演示文稿
- 若需要 Word/PPT，明确告知转换方案

3. 新增金字塔原理、MECE 原则、So What 测试

**完整代码**:
```markdown
# 咨询写作核心规范模块

## 一、咨询写作风格原则

### 1.1 金字塔原理
- 结论先行：先说最重要的结论
- 逐层展开：从总体到细节
- 分组归纳：相似内容归类

### 1.2 MECE 原则
- Mutually Exclusive：分类互斥
- Collectively Exhaustive：完全穷尽
- 应用场景：问题分解、方案分类、市场细分

### 1.3 So What 测试
每个发现必须回答三个问题：
1. 这意味着什么？（含义）
2. 为什么重要？（影响）
3. 应该做什么？（行动）

## 二、去AI化语言规范（保留并调整）

### 2.1 禁用表达
1. 机械过渡词：首先、其次、最后、此外、另外
2. 空洞强调句：值得注意的是、需要指出的是
3. 无数据支撑的形容词：非常、极其、十分

### 2.2 允许的专业表达
1. 明确观点：我们认为、建议、应当、必须
2. 优先级标注：高优先级、关键、次要
3. 行动导向：立即执行、短期内、长期规划

### 2.3 数据驱动表达
- 用"增长 23%"替代"大幅增长"
- 用"市场份额从 15% 提升至 28%"替代"显著提升"
- 标注数据来源和时间："根据 Gartner 2025 年报告"

## 三、段落构建规则

### 3.1 标准段落结构
1. 主题句：核心结论（1 句）
2. 支撑句：数据、案例、分析（2-4 句）
3. 行动句：建议或下一步（1 句）

### 3.2 段落长度
- 中文：100-200 字
- 英文：80-150 词
- 避免超过 250 字的长段落

## 四、列表使用规范

### 4.1 适合使用列表的场景
- 关键发现（3-5 条）
- 建议清单（带优先级）
- 实施步骤
- 风险清单

### 4.2 列表格式要求
- 每条独立完整
- 并列关系清晰
- 可加粗关键词
- 可标注优先级

## 五、三轮质量检查

### 第一轮：逻辑检查
1. 结论是否有数据支撑
2. 分类是否符合 MECE
3. 每个发现是否通过 So What 测试

### 第二轮：语言检查
1. 是否有机械过渡词
2. 是否有空洞形容词
3. 数据是否标注来源

### 第三轮：排版检查
1. 段落间是否空一行
2. 列表格式是否统一
3. 加粗是否适度
```

# 执行摘要写作模块

## 一、执行摘要的作用

执行摘要是咨询报告最重要的部分，通常是决策者唯一会完整阅读的内容。

**目标读者**: CEO、高管、决策者
**阅读时间**: 2-3 分钟

## 二、标准结构

### 2.1 四段式结构

1. **背景与目标**（2-3 句）

   - 研究目标

2. **关键发现**（3-5 条）
   - 最重要的洞察
   - 用数据支撑
   - 每条 1-2 句

3. **核心建议**（3-5 条）
   - 可执行的建议
   - 标注优先级
   - 预期影响

4. **下一步行动**（2-3 句）
   - 立即行动项
   - 时间线
   - 资源需求

## 三、写作原则

1. **结论先行**: 第一句就说最重要的结论
2. **数据驱动**: 每个发现用数字支撑
3. **面向行动**: 建议具体可执行
4. **独立完整**: 不读正文也能理解

## 四、质量检查清单

- [ ] 是否在 1-2 页内
- [ ] 是否包含 3-5 个关键发现
- [ ] 每个发现是否有数据支撑
- [ ] 建议是否标注优先级
- [ ] 是否独立完整

## 第七部分：关键设计决策

### 决策 1：保留 plan/ 目录管理
**理由**: 这是原 skill 的核心优势，咨询项目同样需要上下文管理

### 决策 2：调整写作风格但保留去AI化
**理由**: 咨询报告需要专业性，但仍要避免机械化表达

### 决策 3：新增商业分析框架
**理由**: 咨询报告需要结构化分析工具（SWOT、Porter 等）

### 决策 4：支持 PPT 生成
**理由**: 咨询项目通常需要演示文稿，这是学术论文不需要的

### 决策 5：简化环境配置
**理由**: 咨询场景主要用 Python 做数据分析和图表，不需要复杂的学术环境
